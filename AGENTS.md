## 📋 기본 규칙

### 1. 작업 시작 전
- **이 파일(Agents.md)을 먼저 읽을 것**
- **{Spec.md}를 참조**하여 프로젝트 구조 파악 (또는 유사한 설계 문서)
- 작업 전에 사용자에게 **진행 여부를 물어볼 것**
- 긍정적인 답을 받아야 작업 시작

### 2. 코드 작성 규칙
- **SOLID 원칙 준수**
- **DI(의존성 주입) 패턴 채택**
- **데이터와 렌더링 계층 분리** (Core는 데이터 로직, 프레젠테이션 계층은 시각화)
- 프레임워크 종속 코드는 별도 계층으로 분리 (Unity 등)
- 코어 로직은 프레임워크 의존성을 최소화하여 순수 코드로 작성
- **Surgical Change 원칙**: 요청과 직접 관련 없는 코드는 건드리지 않는다.
  - 인접 코드, 주석, 포맷을 "개선"하지 않는다.
  - 고장 나지 않은 것을 리팩토링하지 않는다.
  - 기존 스타일이 마음에 들지 않아도 현재 스타일을 따른다.
  - 모든 변경 라인은 사용자 요청에서 직접 추적 가능해야 한다.
  - 단, 내 변경으로 인해 사용되지 않는 import/변수/함수는 정리한다.

### 3. Git 규칙
#### 커밋 규칙
- 작업 항목 완료 시 **즉시 Commit**
- 커밋 메시지: 간결하게 (한 줄)
- 예: `Add: FeatureName - 간단한 설명`

#### 브랜치 전략
```text
main              ← 항상 실행 가능해야 함
  feature/xxx     ← 기능별 브랜치에서 작업
  fix/xxx         ← 버그 수정 브랜치
```
- `main` 직접 수정 금지
- 기능/수정별 브랜치에서 작업 후 병합

#### 병합 규칙
- 병합 전 최소 1명이 실행 확인
- 충돌 발생 시 코어 기능 우선 보존
- 병합 후 즉시 main 실행 가능 상태 확인

### 4. 사용 금지 도구
- **unity-cli 사용 금지** — Unity Editor를 CLI로 제어하는 도구. 현재 프로젝트에서 작동하지 않음.
- CLI 도구 대신 **파일을 직접 읽어서** Unity 정보를 확인한다:
  - `*.unity` (씬 파일) → YAML 형식. 하이라키 구조, GameObject, 컴포넌트 연결 확인 가능
  - `*.prefab` (프리팹 파일) → YAML 형식. 프리팹 구조, 컴포넌트, 프로퍼티 값 확인 가능
  - `*.asset` (ScriptableObject 등) → YAML 형식. SO 인스턴스의 필드 값 확인 가능
  - `*.meta` → GUID, 파일 임포트 설정 확인 가능
  - 예: 씬의 하이라키 구조를 보려면 `Assets/_Project/Scenes/Prototype.unity` 파일을 읽음
  - 예: 프리팹 구성을 보려면 `Assets/_Project/Prefabs/...prefab` 파일을 읽음
- 구체적인 금지 사항은 Spec.md 또는 프로젝트 문서 참조

### 5. 문서화 규칙
- 마크다운(.md) 형식 사용
- 상태별 문서 분리:
  - `progress/` - 진행 중 문서
  - `reports/` - 완료 보고서
  - `plans/` - 계획 문서

## 🔍 의견/개선 제안 검증 절차

### 1. 원칙
- 의견이나 개선안 제시 전 **반드시 한 번 더 검증**할 것
- 검증되지 않은 의견은 제시하지 않음

### 2. 검증 절차
```
[의견/개선안 도출]
    ↓
[관련 코드/문서 재확인]
    ↓
[틀린 대답 여부 체크] ← 확실하지 않은 내용은 표기 포맷 사용
    ↓
[검증 완료 후 제시]
```

### 3. 검증 기준
| 항목 | 설명 |
|------|------|
| 사실 확인 | 관련 코드/문서를 다시 읽고 확인했는가? |
| 오류 여부 | 틀린 대답이나 오해를 불러일으킬 내용이 없는가? |
| 근거 명확성 | 제시하는 내용에 충분한 근거가 있는가? |
| 불확실성 표기 | 확실하지 않으면 정해진 포맷으로 명시 |

### 4. 불확실성 표기 포맷

| 표기 | 의미 | 사용 예시 |
|------|------|-----------|
| `[추정]` | 문서/코드에서 직접 확인하지 못하고 유추한 내용 | "[추정] 이 함수는 ~할 것으로 보입니다" |
| `[확인필요]` | 추가 정보나 사용자 확인이 필요한 사항 | "[확인필요] 이 브랜치의 병합 상태를 확인해야 합니다" |
| `[가정]` | 명시적으로 전제한 조건 | "[가정] 사용자가 최신 main을 기준으로 작업한다고 가정합니다" |

### 라이선스 퀵 가이드

| License | Use? | Rule |
|---|---|---|
| CC0 | Yes | 상업 사용/수정 가능. 기록 권장. |
| CC-BY | Yes | 상업 사용 가능. 크레딧 필수. |
| MIT | Yes | 코드/라이브러리에 적합. 라이선스 고지 권장. |
| OFL | Yes | 폰트에 적합. 라이선스 파일 유지 권장. |
| Unity Asset Store EULA | Usually Yes | 게임 포함 가능. 단독 재배포 금지. |
| CC-BY-NC | No | 상업 사용 금지. |
| CC-BY-ND | No | 수정 금지. 게임 에셋에 부적합. |
| GPL | Avoid | 프로젝트 공개 의무 등 관리가 어려움. |
| Unknown | No | 라이선스 불명확. 사용 금지. |
